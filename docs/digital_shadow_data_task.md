# ТЗ для студентов: сбор данных для Digital Shadow

**Цель.** Собрать датасет для обучения и демо детектора Digital Shadow (OSINT/DarkNet):
контрабанда (вейпы/алкоголь/наркотики), вербовка дропов, подозрительные криптокошельки,
утечки баз данных РК. Каждый кейс прогоняется через пайплайн (`apps/digital_shadow`) и должен
зажигать ожидаемые риск-сигналы и категорию.

---

## 0. Правила (ОБЯЗАТЕЛЬНО — нарушение = снятие задачи)

Это OSINT для **защиты**, а не участие в незаконном. Жёстко:

- **Только публично доступный текст/метаданные.** Не входить в закрытые группы, не авторизовываться
  ради доступа, не обходить блокировки.
- **Никаких контактов и покупок.** Не писать продавцам, не «уточнять цену», не заказывать, не переводить деньги.
- **Не скачивать и не хранить реальные ПДн** из утечек/дампов (ФИО, ИИН, карты, телефоны, пароли).
  Фиксируем только **факт** объявления + маскированные индикаторы. ИИН/карты/телефоны → маска
  (`************`, `4400****1234`, `+7 7XX XXX-XX-XX`).
- **Наркотики и реальный DarkNet — НЕ краулить.** Для этих категорий используем **синтетику**
  (генератор) и **публичные исследовательские датасеты**, а не реальные площадки.
- Кошельки — публичные строки адресов (из открытых abuse-репортов) ок; реальные ПДн владельца — нет.
- При сомнении «можно ли это собирать» — спросить координатора, не собирать.

> Принцип системы (§0): мы выявляем риск-сигналы и передаём на ручную проверку, не выносим обвинение.

---

## 1. Что система принимает на вход

Один элемент = **ShadowItem** (см. `apps/digital_shadow/schemas.py`):

| Поле | Обяз. | Что |
|---|---|---|
| `id` | да | уникальный, префикс по категории (`vape_001`, `drop_012`…) |
| `source_type` | да | `clearweb` · `darknet` · `paste` |
| `source_url` | нет | ссылка/площадка (для графа-источника); для darknet — выдуманная/`.onion` |
| `platform` | нет | olx / instagram / telegram / forum / paste-сайт |
| `title` | нет | заголовок объявления |
| `text` | да | видимый текст объявления/листинга/поста |
| `language` | нет | `ru` / `kk` / `en` |

Подаём либо по одному (`POST /shadow/analyze`), либо пачкой в JSONL (см. §4).

## 2. Категории, что искать и ГДЕ (по приоритету легальности)

Для каждой категории — где искать, поисковые запросы (ru/kk) и ожидаемые сигналы.

### 2.1 Контрабанда вейпов — `contraband_vape`  *(clearweb, легально)*
- **Где:** OLX.kz, Instagram, Telegram-каналы продаж, объявления, маркетплейсы.
- **Запросы:** «вейп оптом», «elf bar оптом», «жидкости без акциз», «электронные сигареты без документов»,
  «айкос оптом»; kk: «акцизсіз сұйықтық», «вейп көтерме».
- **Ожидаемо:** `contraband_keyword`, `bulk_sale`, `no_kyc`, `encrypted_contact` (увод в telegram).

### 2.2 Нелегальный алкоголь — `contraband_alcohol`  *(clearweb, легально)*
- **Где:** объявления, Telegram, соцсети.
- **Запросы:** «алкоголь без акциз», «спирт оптом», «паленый алкоголь», «арақ көтерме».
- **Ожидаемо:** `contraband_keyword`, `bulk_sale`, `no_kyc`.

### 2.3 Вербовка дропов — `drop_recruitment`  *(clearweb, легально, ценно)*
- **Где:** сайты «подработка», Telegram-каналы вакансий, VK, объявления.
- **Запросы:** «ищем дропов», «карта в аренду», «приём переводов за процент», «обнал»,
  «выгодная подработка финансы», «нужны карты»; kk: «карта жалға».
- **Ожидаемо:** `drop_recruitment`, `encrypted_contact`, часто `crypto_only_payment`, `money_mule_request`.

### 2.4 Подозрительные криптокошельки — `suspicious_crypto`  *(публичные abuse-репорты)*
- **Где:** **публичные** базы жалоб: chainabuse.com, bitcoinabuse / «scam wallet report», открытые
  threat-репорты. Берём адрес + текст жалобы (контекст), НЕ персональные данные.
- **Что собрать:** текст вида «кошелёк X в скам-схеме/вымогательстве», адрес BTC/ETH/USDT(TRC20).
- **Ожидаемо:** `crypto_wallet_found`, при наличии слов про отмыв — `mixer_or_tumbler`;
  адреса из публичных «плохих» списков можно занести в `crypto_risk._BAD_WALLETS` → `bad_crypto_wallet`.

### 2.5 Утечки БД РК — `kz_data_leak`  *(метаданные/факт, БЕЗ ПДн)*
- **Где:** объявления о продаже баз на форумах/paste-сайтах (текст листинга), публичные
  breach-сводки. **НЕ скачивать дамп, не сохранять ПДн.**
- **Запросы:** «продам базу РК», «слив базы Казахстан», «пробив», «дамп база», «leaked database kazakhstan».
- **Что собрать:** только текст объявления (что продают, цена в крипте, контакт). ИИН/телефоны в тексте → замаскировать.
- **Ожидаемо:** `kz_data_leak`, `iin_dump_mention` (если упомянуты ИИН/ПДн), `crypto_only_payment`, `encrypted_contact`.

### 2.6 Наркотики — `drug_trafficking`  *(СИНТЕТИКА + research-датасеты, НЕ краулить)*
- **Где:** **не собирать с реальных площадок.** Сгенерировать синтетические листинги (по образцу
  `collectors/darknet_mock.py`) и/или взять обезличенные примеры из публичных research-корпусов о даркнете.
- **Ожидаемо:** `drug_slang`, `darknet_listing`, `crypto_only_payment`, `encrypted_contact`.

> Контрпримеры (обязательно!): легальные объявления — обычная продажа вейпа **с акцизом/документами**,
> легальная вакансия, обычный обмен крипты на бирже. Нужны, чтобы детектор не красил всё в «urgent».

## 3. Объём (минимум на демо)
- `contraband_vape` — 20 · `contraband_alcohol` — 15 · `drop_recruitment` — 25 (самая ценная, реальная)

- баланс языков: добавить ≥20% kk.

## 4. Формат сдачи (JSONL)
Файл по категории `data/shadow/<category>.jsonl` или общий `data/processed/shadow_demo.jsonl`.
Одна строка = один JSON. Поля §1 + «эталон» для проверки (`gold_category`, `gold_signals`).

```json
{"id":"drop_001","source_type":"clearweb","platform":"telegram","language":"ru","title":"Подработка","text":"Ищем дропов: карта в аренду за %, приём переводов и снять. Telegram @work_fast, оплата USDT.","gold_category":"drop_recruitment","gold_signals":["drop_recruitment","encrypted_contact","crypto_only_payment"]}
```
```json
{"id":"legit_001","source_type":"clearweb","platform":"olx","language":"ru","title":"Вейп","text":"Продам вейп, чек и гарантия, продажа только 21+, оригинал.","gold_category":"unknown","gold_signals":[]}
```

Правила: `id` уникальный с префиксом категории; реквизиты — только маски/выдумка (§0);
для darknet/drugs `source_type:"darknet"` и выдуманный `source_url`.

## 4.1 Стартовый датасет (уже есть — расширяйте его)
В репозитории лежит **синтетический seed** `data/shadow/seed.jsonl` (55 размеченных кейсов по всем
категориям + legit-контрпримеры), сгенерированный `apps/digital_shadow/seed_data.py`. Это образец
формата и «скелет», на который вы добавляете СВОИ кейсы.

```bash
make shadow-seed                 # пересобрать синтетический seed (если меняли генератор)
make shadow-eval                 # прогнать seed через пайплайн + метрики (точность/покрытие)
make shadow-eval DATA=data/shadow/my_collected.jsonl   # оценить свой файл
```

Как дополнять:
1. Добавляйте строки в `data/shadow/seed.jsonl` (или свой файл того же формата) — реальные clearweb-кейсы
   (вейпы/дропы), публичные abuse-репорты (крипто), метаданные листингов утечек.
2. Прогоняйте `make shadow-eval` — смотрите, где категория/сигналы не совпали с `gold_*`.
3. Если детектор пропускает — расширяйте лексикон/правила в `apps/digital_shadow/taxonomy.py`
   (новые слова, новые категории) и повторяйте, пока точность не вырастет.

### Авто-генерация через LLM (быстро нарастить объём)
Если поднят vLLM (`bash scripts/run_vllm.sh`), модель сама пишет разнообразные синтетические
листинги по категориям, и каждый **самопроверяется** прогоном через пайплайн — в файл попадают
только корректно классифицированные (gold-метки гарантированно верные).

```bash
make shadow-gen SCALE=0.3            # → data/shadow/llm_gen.jsonl (увеличивайте SCALE для объёма)
make shadow-eval DATA=data/shadow/llm_gen.jsonl
# объединить синтетику с seed и оценить всё вместе:
cat data/shadow/seed.jsonl data/shadow/llm_gen.jsonl > data/shadow/all.jsonl
make shadow-eval DATA=data/shadow/all.jsonl
```
Низкий выход по категории (напр. kk-вейп, крипто) = детектор её слабо ловит → это сигнал
расширить `taxonomy.py` (п.3) и перегенерировать. LLM-синтетика НЕ заменяет реальные данные —
это объём для baseline; реальные clearweb/abuse-кейсы по §2 всё равно нужны.

## 5. Как прогнать цикл и проверить
Поднять сервис: `docker compose -f infra/docker-compose.yml up -d qdrant neo4j postgres && make shadow`,
фронт — `make shadow-front` (http://localhost:8091).

```bash
# по одному — через UI (вкладка «Анализ») или curl:
curl -X POST localhost:8090/shadow/analyze -H 'Content-Type: application/json' \
  -d '{"id":"drop_001","source_type":"clearweb","text":"Ищем дропов, карта в аренду за %, telegram @x, оплата USDT"}'
```
Проверять по каждому кейсу: совпали ли `category` и `signals` с `gold_*`; находки видны во вкладке
«Находки» (Postgres) и в «Теневом графе» (общий Neo4j — повторяющиеся кошельки/домены связываются).

**Боевой сбор (не eval, а загрузка реальных источников в граф+БД):** когда насобирали листинги в
JSONL, прогоняйте их коллектором — он сразу пишет в общий граф и `shadow_findings`:
```bash
make shadow-collect ARGS="--file data/shadow/inbox.jsonl"     # ваш экспорт листингов
make shadow-collect ARGS="--rss https://site/feed.xml"        # RSS-лента
# страница / Tor (.onion): --url ... [--proxy socks5h://127.0.0.1:9050]
```

**ML-классификатор (опц.):** на собранном датасете можно обучить категорийный классификатор —
`make shadow-train` (macro-F1 ≈ 0.98 на текущих 326). Он подстраховывает правила, когда они дают
`unknown` (включается `SHADOW_ML=1`).

## 6. Чек-лист приёмки
- [ ] объёмы по §3 набраны, есть ≥20% kk и **legit-контрпримеры**.
- [ ] у каждого кейса проставлены `gold_category` + `gold_signals`.
- [ ] §0 соблюдён: только публичный текст, без контактов/покупок, без реальных ПДн (всё маскировано).
- [ ] наркотики/даркнет — синтетика/research, не краулинг реальных площадок.
- [ ] прогон через `/shadow/analyze`: ≥80% кейсов дают ожидаемую категорию.
- [ ] видно различие: contraband/drop/leak → high/urgent, legit → low.

Таксономия и сигналы — `apps/digital_shadow/taxonomy.py`; общая архитектура — [`two_products.md`](two_products.md).
